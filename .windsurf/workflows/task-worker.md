Ты — Cascade. Работай по дереву задач MCP в ТЕКУЩЕМ ПРОЕКТЕ (по умолчанию: "mcp"), источник задач: @mcp:mcp-task-knowledge-(gpu).
Всегда действуй по шагам ниже, не уходи в дебри, не выполняй разрушительные команды без явного подтверждения.

=== ПРАВИЛА КОНТЕКСТА ===
1. MCP-трекер = единственный источник правды (контекст текущего проекта).
   - Никогда не храни планы/подзадачи только во внутреннем диалоге.
   - Все планы и декомпозицию фиксируй через MCP:
     - mcp3_tasks_create — создание подзадач
     - mcp3_tasks_update — изменение статуса
     - комментарии к задаче — краткие планы/заметки.
2. После каждого инкремента делай синхронизацию (mcp3_tasks_tree).
3. Работа должна быть продолжабельна в новом сеансе без потери контекста.

=== ПРАВИЛА ВЫБОРА ЗАДАЧИ ===
- Приоритет: in_progress > pending > подзадачи сверху вниз.
- Если выбрана родительская → выбери ближайшую подзадачу к исполнению.
- Definition of Done (DoD): критерий полной готовности.
- Definition of Sufficient (DoS): минимальный полезный результат (снятый риск или работающий кусок с тестом).

=== СТРОГИЙ РЕЖИМ АГЕНТА (без пост‑правок выгрузки) ===
Обязательные требования при создании/обновлении задач и знаний — несоблюдение недопустимо.

1) Задачи (mcp3_tasks_create / update)
- Title обязан начинаться с `MCP-XXX-###` (регекс `^MCP-[A-Z]{3}-[0-9]{3,}`), затем короткое описание.
- Должны быть заданы: `parentId` (кроме корня), приоритет/теги компонента (`EXEC|SCHED|API|…`).
- Запрещено создавать задачи без родителя, если родитель логически требуется (иерархия не плоская).
- При update статус только из множества: `pending|in_progress|blocked|completed|closed`.

2) Знания (mcp3_knowledge_create / update)
- Должны иметь корректный `parentId`, указывающий на раздел/компонент дерева знаний.
- Title рекомендуется начинать с ID задачи/инициативы (`MCP-XXX-### — …`) при наличии связи.
- Перед созданием выполнять поиск дублей (`mcp3_search_knowledge`).

3) Проверка дерева после каждого инкремента
- Выполнять `mcp3_tasks_tree` и убеждаться в отсутствии циклов/осиротевших узлов.
- Если нарушена иерархия — сначала исправить структуру (reparent/move_subtree), затем продолжать работу.

4) Шаблоны (frontmatter) при генерации контента
- Для задач и знаний использовать шаблоны из раздела «Стандарты оформления (Frontmatter)» ниже.
- Отказывать в создании заметки, если невозможно заполнить обязательные поля.

=== АНТИ-ЗАЦИКЛИВАНИЕ ===
- TIMEBOX_MIN = 20 минут на инкремент
- MAX_INCREMENTS_PER_TASK = 2 подряд
- SAFE_DIFF_LIMIT = ~200 строк diff
- NO_PROGRESS_GUARD = если нет осмысленных изменений → переключись

=== ЧЁТКАЯ ПОДЗАДАЧА (Definition of Clear Task) ===
Каждая подзадача обязана содержать:
- Название: кратко и конкретно.
- Описание:
  - Цель: зачем эта подзадача нужна.
  - DoD: конкретный результат на выходе.
  - Ограничения/допущения (если есть).

Пример:
Название: "Добавить unit-тест для edge-case X"
Описание:
- Цель: убедиться, что функция Y корректно обрабатывает X.
- DoD: тест падает на старом коде и зелёный после фикса.
- Ограничения: только unit-тест, без интеграции.

=== ИНСТРУМЕНТАЛЬНЫЙ ЦИКЛ ===
1) Синхронизация задач
   - mcp3_tasks_tree / mcp3_tasks_list
2) Выбор задачи
   - Обоснование выбора
   - Зафиксируй DoD и DoS
3) План инкремента (≤5 пунктов, ≤20 мин, ≤200 строк diff)
4) Выполнение
   - Код через редактор
   - Тесты/линтеры
5) Проверка
   - DoD → готово
   - DoS / таймбокс / max2 → переключение
   - Нет прогресса → переключение
6) Решение о переключении
   - DoD → close
   - DoS/таймбокс/max2 → подзадачи + статус
   - Блокер → status=blocked + подзадача-напоминание
7) Обновление статуса (mcp3_tasks_update)
8) Отчёт: файлы, тесты, что дальше
9) Повторная синхронизация → шаг 1

=== ЗАПРЕТЫ ===
- Не выполняй внешние запросы и не ставь зависимости без подтверждения.
- Не запускай небезопасные команды.
- Не превышай SAFE_DIFF_LIMIT за инкремент.

=== ФИКСАЦИЯ ЗНАНИЙ ===
1. MCP Knowledge-tree = база знаний проекта, единственный источник накопленных правил и решений.
2. После выполнения задачи оцени, появилось ли новое знание:
   - Архитектурное решение или паттерн
   - Неочевидный баг и его обход
   - Конвенция/правило команды
   - Повторяемый сценарий/шаг/best practice
3. Если знание полезно повторно — зафиксируй его в базе знаний:
   - mcp3_knowledge_create(title, content, parentId?, tags?) — создать новый узел
   - mcp3_knowledge_update(id, content|title|tags|parentId) — обновить существующий
   - mcp3_knowledge_list / get — для навигации; link можно фиксировать в описании задачи
4. Структура записи:
   - Название: кратко и конкретно
   - Содержание:
     - Контекст (где применимо)
     - Решение / правило / паттерн
     - Пример (по возможности)
5. Перед созданием нового узла проверяй (mcp3_search_knowledge), не существует ли похожего знания, чтобы не дублировать.
6. Важное правило: не фиксируй очевидное или одноразовое, сохраняй только то, что пригодится повторно.

# ✅ ЧЕК-ЛИСТ CASCADE (Quickref)

Цикл:
1. Синхронизация → mcp3_tasks_tree
2. Выбор задачи → приоритет: in_progress > pending > сверху вниз
3. DoD + DoS зафиксировать
4. План ≤5 пунктов / ≤20 мин / ≤200 строк diff
5. Выполнение → код + тесты + линтер
6. Проверка:
   - DoD → закрыть
   - DoS/таймбокс/max2 → подзадачи, переключиться
   - Нет прогресса → переключиться
7. Обновление статуса в MCP (mcp3_tasks_update)
8. Отчёт: файлы, тесты, что дальше
9. Повторная синхронизация → шаг 1

Правила:
- Нет внутренних планов → всё в MCP.
- Подзадача = цель + DoD + ограничения.
- Блокер → status=blocked + подзадача-напоминание.

# ✅ Чек-лист фиксации знаний

- Есть ли новое знание (паттерн, решение, конвенция, баг/обход)?
- Поискать в базе знаний (mcp3_search_knowledge).
- Если нет — создать запись (mcp3_knowledge_create).
- Структура: Название + Контекст + Решение + Пример.
- При необходимости обновить/связать с задачей (mcp3_knowledge_update / ссылка из задачи).
- Фиксируй только то, что пригодится повторно.

# 🗓 Политика CHANGELOG (структура и оформление)

=== РАЗМЕЩЕНИЕ В ДЕРЕВЕ ===
- Путь: `Knowledge/CHANGELOG/`.
- Иерархия:
  - `Knowledge/CHANGELOG/INDEX.md` — оглавление и ссылки на дни.
  - `Knowledge/CHANGELOG/YYYY/INDEX.md` — годовой индекс.
  - `Knowledge/CHANGELOG/YYYY/MM/INDEX.md` — месячный индекс.
  - `Knowledge/CHANGELOG/YYYY/MM/CHANGELOG — YYYY‑MM‑DD.md` — дневная запись (1 запись на дату).
- Каждый каталог обязан иметь `INDEX.md` с ссылками на дочерние узлы.

=== НЕЙМИНГ ФАЙЛОВ ===
- Единый формат для дневной записи: `CHANGELOG — YYYY‑MM‑DD.md`.
- Язык — русский, формат даты ISO.

=== FRONTMATTER (обязательно) ===
```yaml
---
type: changelog
title: "CHANGELOG — 2025-08-26"
status: active
tags: ["changelog"]
updated: 2025-08-26
parentId: <uuid раздела CHANGELOG или null>
links: [
  { task: "MCP-EXEC-312", status: "completed" },
  { task: "MCP-DOC-301", status: "in_progress" }
]
---
```

=== СОДЕРЖАНИЕ ЗАПИСИ (шаблон) ===
```markdown
# Итоги дня
- Кратко: 1–3 предложения, какой полезный результат получен.

## Изменения
- [MCP-EXEC-312] Оптимизирован чекпойнтер Postgres (файлы: `service/executor/checkpointer_postgres.go`).
- [MCP-DOC-301] Добавлены стандарты frontmatter.

## Затронутые области/компоненты
- EXEC, DOC, API …

## Что дальше
- Следующие шаги/подзадачи.

## Ссылки
- Задачи: MCP-EXEC-312, MCP-DOC-301
- PR/коммиты/артефакты.
```

=== ПРАВИЛА ВЕДЕНИЯ ===
- Одна запись на день. При повторном инкременте — обновлять текущую дневную запись (поле `updated`).
- Все закрытые задачи за день перечислять в "Изменения" с указанием файлов.
- Для незавершённых работ отмечать статус и что блокирует.
- Использовать ссылки на задачи (`MCP-XXX-###`) и на файлы в репо.
- Без "воды": конкретные результаты, файлы, что дальше.

=== ИНДЕКСАЦИЯ ===
- Годовой/месячный индексы содержат список ссылок на вложенные записи в обратном хронологическом порядке.
- Обновлять индексы автоматически при добавлении дневной записи.

=== АВТОМАТИЗАЦИЯ ДЛЯ АГЕНТА ===
- При завершении инкремента:
  1) Найти/создать дневную запись за сегодня.
  2) Сформировать секции по шаблону (минимум: Итоги дня, Изменения, Что дальше).
  3) Проставить `updated` и список `links` на затронутые задачи с их статусом.
  4) Обновить месячный и годовой индексы.
- При отсутствии полезных изменений — запись не создавать.

=== ПРАВИЛО НЕПЕРЕЗАПИСИ (append‑only) ДЛЯ ОДНОЙ ДАТЫ ===
- Никогда не перезаписывать файл дня `Knowledge/CHANGELOG/YYYY/MM/CHANGELOG — YYYY‑MM‑DD.md`.
- Стратегия: только добавление инкрементальных блоков в конец файла.
- Формат инкремента:
  - Заголовок: `## Инкремент YYYY‑MM‑DD HH:MM` (локальное время).
  - Маркеры идемпотентности (ISO 8601 с зоной):
    - `<!-- BEGIN INCR: 2025-08-26T08:09:44+03:00 -->` и `<!-- END INCR: 2025-08-26T08:09:44+03:00 -->` вокруг блока.
    - Перед добавлением искать `BEGIN INCR` с тем же timestamp — если найден, обновить блок, а не дублировать.
- Дополнительно:
  - Поле `updated` в frontmatter файла дня всегда отражает время последнего инкремента.
  - Годовой/месячный индексы не дублируют ссылку на один и тот же дневной файл, независимо от числа инкрементов.
- Если нужно зафиксировать отдельный большой релиз в тот же день — добавлять подзаголовок `### Релиз: <имя>` внутри последнего инкремента, а не новый файл.

# 📐 Политика иерархии KB и Jira‑нейминг

=== ИЕРАРХИЯ KB (дерево по компонентам/модулям) ===
   - Корень KB → разделы (Discovery, Naming, Process, CHANGELOG) → компоненты.
   - Под каждый компонент/модуль создаётся узел‑родитель, под ним — знания, гайды, ADR, примеры.
2) Привязка к репозиторию
   - Если есть явная карта компонентов → использовать её.
   - Иначе отражать структуру каталогов верхнего уровня: `service/`, `infra/`, `entities/`, `utils/` и т.д.
3) Записи должны иметь корректный parentId (компонент/модуль).
   - Если запись кросс‑компонентная → размещать в ближайшем общем родителе (например, `Process`, `Discovery`) и тегировать компонентами.

=== JIRA‑ПОДОБНЫЙ НЕЙМИНГ ДЛЯ ЗАДАЧ И ЗНАНИЙ ===
1) Формат идентификатора: `MCP-XXX-###` (пример: `MCP-EXEC-312`).
   - Регекс: `^MCP-[A-Z]{3}-[0-9]{3,}$`
2) Задачи
   - Title: начинается с идентификатора, далее короткое описание.
     Пример: `MCP-EXEC-312 Оптимизировать чекпойнтер Postgres`
   - Tags: включают код компонента (`EXEC`, `SCHED`, `API` и т.п.) и тип (`bug`, `feat`, `doc`).
3) Знания (KB)
   - Рекомендуется начинать Title с идентификатора, если знание привязано к задаче/инициативе.
     Пример: `MCP-EXEC-312 — Чеклист оптимизации чекпойнтера`
   - Tags: включают код компонента и раздел (`KB`, `ADR`, `DOC`, `pattern`, `bugfix`).

=== ЧЕК‑ЛИСТ ВАЛИДАЦИИ ПЕРЕД СОЗДАНИЕМ ===
- Указан ли корректный parentId (компонент/модуль)?
- Соответствует ли Title регексу `^MCP-[A-Z]{3}-[0-9]{3,}` в начале?
- Проставлены ли теги компонента и типа?

- Ввести предполагаемый ID и Title → сопоставить с регексом.
- При несоответствии — скорректировать до `MCP-XXX-### Описание`.

// turbo
Быстрая проверка расположения в дереве
- Определить компонент/модуль по карте компонентов или пути файла.
- Убедиться, что parentId указывает на узел компонента; иначе — перенести/исправить parentId.

# 📚 Стандарты оформления (Frontmatter) и структура в Obsidian

=== ОБЩЕЕ ===
- Все заметки (и задачи, и знания) обязаны содержать корректный YAML‑frontmatter на первой строке файла.
- Обязательные поля: `title`, `status`, `tags`, `id` (для задач), `parentId` (если есть родитель), `updated` (ISO‑дата), `type`.
- Статусы задач: `pending | in_progress | blocked | completed | closed`.

=== ШАБЛОН — Задача (Task) ===

```yaml
---
type: task
id: MCP-XXX-###
title: MCP-XXX-### Короткое название
status: pending
parentId: <uuid или null>
tags: ["task", "<COMP>", "<feat|bug|doc>"]
updated: 2025-01-01
---
# Цель
# DoD
# Ограничения
# Ссылки/контекст
```

=== ШАБЛОН — Знание (Knowledge) ===

```yaml
---
type: knowledge
title: MCP-XXX-### — Краткое название знания
status: active
parentId: <uuid узла-компонента или раздела>
tags: ["knowledge", "<COMP>", "<KB|ADR|DOC|pattern|bugfix>"]
updated: 2025-01-01
---
## Контекст
## Решение / Правило
## Пример
```

=== ВАЛИДАЦИЯ ДО/ПОСЛЕ ЭКСПОРТА ===
- Проверить, что каждая задача имеет корректный `parentId` и входит в `mcp3_tasks_tree` без циклов.
- Проверить наличие обязательных полей в YAML‑шапке.
- Обеспечить наличие `INDEX.md` в каждом каталоге дерева.
